What This Project Is

A production-quality, bare-metal audio driver written in pure C. It drives the I2S2 peripheral of an STM32F407 microcontroller, outputs a continuous 440 Hz tone via DMA double-buffering, and converts PDM microphone input to PCM using a CIC + FIR decimation pipeline — all without an OS, HAL, or standard library.

🎯 Goal

Demonstrate real-time audio driver skills: register-level peripheral control, DMA, interrupt-safe data structures, and DSP — exactly what companies like Qualcomm, ARM, and Apple test for.

⚙️ Target Hardware

STM32F407VGT6 — ARM Cortex-M4F, 168 MHz, 1 MB Flash, 128 KB SRAM. Simulated in QEMU using the netduinoplus2 machine model.

📐 Key Constraints

No dynamic allocation, no OS, no libc. Every timing guarantee is met via DMA circular mode + ping-pong buffers. Code runs on Cortex-M0 too (Q15 not float).

What This Project Actually Does

A plain-language walkthrough for anyone who doesn't want to parse register maps and DSP math first. If you already know embedded systems, the Overview and Concepts & Patterns tabs go deeper.

The one-sentence version

This code turns a microcontroller into a tiny, dedicated audio device — it listens to a digital microphone, cleans up the signal, and continuously plays a tone out of a speaker, doing all of the timing-critical work in hardware so the CPU is barely involved.

Why "bare-metal" is the interesting part

Almost every audio app you've used sits on top of an operating system, a hardware abstraction layer (HAL), and a standard library. This project has none of that. There's no Linux, no RTOS, no printf, no malloc — just C code talking directly to memory addresses that happen to be physical hardware registers on an STM32F407 chip. That's what "bare-metal" means: the program is the operating system.

It's built and proven without needing to own the physical board — everything compiles and runs against a QEMU simulation of the exact chip (netduinoplus2), and the DSP logic is separately verified with 27 native unit tests that run on a regular laptop.

The journey of one sample, in plain English

1. Microphone → bits

A PDM microphone doesn't send nice numbers — it sends a raw stream of 1s and 0s at 3 million bits per second. Think of it as a firehose of noisy Morse code.

2. Bits → numbers

A "CIC filter" — cheap hardware-friendly math using only addition and subtraction — turns those 3 million bits/sec into 48,000 real numbers/sec (audio samples), the same rate as a normal digital recording.

3. Cleanup

A second filter (FIR) smooths out artifacts the first filter left behind, the same way a moving average smooths a noisy stock chart.

4. Handoff to the speaker

Two buffers take turns: while one is being played out to the speaker, the other is being filled with the next batch of samples — so there's never a gap or a collision.

5. Out to hardware

A DMA engine — a small helper circuit built into the chip — physically moves the samples to the audio output on its own, so the main processor isn't stuck babysitting every single sample.

Concepts & Design Patterns

Every major technique used in this driver — explained bottom-up with analogies.

1. Memory-Mapped I/O (MMIO)

On ARM Cortex-M, peripherals (I2S, DMA, GPIO) are not separate chips — they're just memory addresses. Writing 0x1 to address 0x40003900 literally enables the I2S peripheral.

/* From i2s.h — the peripheral IS these addresses */
#define I2S2_BASE    0x40003800UL
#define I2S2_CR1     (I2S2_BASE + 0x00)  /* Control register 1 */
#define I2S2_SR      (I2S2_BASE + 0x08)  /* Status register    */
#define I2S2_DR      (I2S2_BASE + 0x0C)  /* Data register      */

/* Writing a sample to the hardware: */
*((volatile uint32_t*)(I2S2_BASE + 0x0C)) = sample;
Analogy

Think of a hotel with numbered rooms. Room 1004 is the "send audio data" room. You walk in and leave a note (write to the address). The hotel (hardware) reads the note and acts on it. The room doesn't store things — it triggers hardware action.

Why volatile? Without volatile, the compiler might optimize away the write ("nobody reads this variable") or reorder it. volatile forces every access to go through the actual memory bus — critical when the "memory" is hardware.

2. The volatile Keyword

volatile uint32_t *reg = (volatile uint32_t*)0x40003808;

/* Without volatile — compiler might eliminate this loop: */
while (*reg & FLAG) {}  /* "reg never changes in C code!" → optimized to infinite loop or removed */

/* With volatile — compiler MUST re-read reg every iteration: */
while (*(volatile uint32_t*)0x40003808 & FLAG) {}  /* correct — hardware changes it */

The status register changes because hardware changes it, not because your C code does. The compiler cannot see hardware — so you must tell it to re-read every time.

3. DMA — Direct Memory Access

DMA is a hardware bus master that can copy data between memory and peripherals without CPU involvement.

/* Conceptual DMA setup (from i2s.c): */
DMA_SxCR  = DIRECTION_MEM_TO_PERIPH   /* memory → I2S data register */
           | CIRCULAR_MODE             /* auto-restart at buffer end */
           | MINC                      /* increment memory address   */
           | DATA_SIZE_16BIT;

DMA_SxNDTR = AUDIO_BUF_SIZE;          /* number of transfers */
DMA_SxM0AR = (uint32_t)audio_buffer;  /* source address      */
DMA_SxPAR  = I2S2_DR;                 /* destination: I2S DR */

Analogy

CPU-driven I/O = you carry each box yourself, one at a time, between warehouse and truck. DMA = you hire a forklift operator — you say "move these 512 boxes," go do other work, and the operator calls you when done.

4. Ping-Pong (Double) Buffering

Two buffers, each large enough for one DMA burst. DMA always reads from the "active" buffer while the CPU fills the "inactive" one. At the ISR, they swap roles.

typedef struct {
    int16_t ping[AUDIO_BUF_SIZE];  /* 512 stereo frames = 1024 int16 */
    int16_t pong[AUDIO_BUF_SIZE];
    uint8_t active;                  /* 0=DMA reads ping, 1=DMA reads pong */
    uint32_t underruns;              /* diagnostic counter */
} AudioBuf;

/* Swap in ISR — single XOR, atomic on ARM: */
buf->active ^= 1;

/* CPU always writes the inactive half: */
int16_t *write = (buf->active == 0) ? buf->pong : buf->ping;

/* DMA always reads the active half: */
int16_t *read  = (buf->active == 0) ? buf->ping : buf->pong;
Analogy

Restaurant with two serving windows: one window serves customers (DMA plays), the other window is being stocked by the kitchen (CPU writes). When the serving window empties, you flip a sign — the stocked window starts serving, the empty one goes back to the kitchen. Customers never wait.

Why no mutex? The swap is a single XOR on a single byte. On ARM Cortex-M, a byte write is atomic — it either happens or it doesn't, never half-done. Since only the ISR swaps, and the main loop only reads active after the swap, there's no race condition.

5. CIC Filter — Cascaded Integrator-Comb

A CIC filter decimates (downsamples) a 1-bit PDM stream using only addition and subtraction — no multiplication. Essential when the input rate (3 MHz) is too fast for multipliers.

/* CIC structure: 3 integrators + 3 combs, decimation = 64 */

/* Integrator (runs every PDM bit): y[n] = y[n-1] + x[n] */
integrator[0] += pdm_bit;           /* accumulate */
integrator[1] += integrator[0];
integrator[2] += integrator[1];

/* Comb (runs every 64th bit): y[n] = x[n] - x[n-1] */
if (++bit_count == 64) {
    bit_count = 0;
    int32_t v = integrator[2];
    comb_out = v - comb_prev;         /* differentiate */
    comb_prev = v;
    /* pass to FIR stage */
}
PropertyValue
Input rate3.072 MHz PDM
Output rate48 kHz PCM
Decimation ratio64×
Filter order3rd order (3 integrators, 3 combs)
Gain64³ = 262,144 (must scale down to int16)
Math operationsAdd/subtract only
Analogy

The CIC integrator is a running sum — like counting votes in real time. The comb stage is "how many more votes since the last checkpoint?" Together they average out the high-frequency noise while keeping the low-frequency signal.

6. FIR Filter — Finite Impulse Response

A 16-tap low-pass FIR runs at 48 kHz to clean up CIC droop and remove frequencies above 19 kHz (half of 48 kHz = Nyquist).

/* 16-tap Hamming-windowed sinc coefficients (Q15, sum ≈ 32767) */
static const int16_t fir_coeffs[16] = {
    -128, -256, 0, 512, 1024, 2048, 4096, 8192,
    8192, 4096, 2048, 1024, 512, 0, -256, -128
};  /* symmetric = linear phase (no phase distortion) */

/* Ring buffer delay line + MAC loop: */
delay[head] = new_sample;
int32_t acc = 0;
for (int i = 0; i < 16; i++) {
    acc += (int32_t)fir_coeffs[i] * delay[(head - i) & 0xF];
}
output = (acc >> 15);  /* Q15 → int16 */
head = (head + 1) & 0xF;
Analogy

An FIR is a weighted moving average. Each output sample is the dot product of the last 16 input samples with the coefficient weights. The symmetric shape means it's "balanced" — no phase shift, just smoothing.

7. Q15 Fixed-Point Arithmetic

Q15 represents fractional numbers in a 16-bit integer. The implicit binary point is after bit 15: range is -1.0 to +0.9999 (i.e., -32768 to +32767).

/* Represent 0.5 in Q15: */
int16_t half = (int16_t)(0.5f * 32767.0f);  /* = 16383 */

/* Multiply two Q15 numbers: result is Q30, shift down to Q15 */
int32_t product = (int32_t)a * b;  /* Q30 */
int16_t result  = (product >> 15); /* Q15 */

/* Sine LUT: 256 entries, full cycle */
lut[i] = (int16_t)(32767.0f * sinf(2.0f * M_PI * i / 256.0f));

8. Vector Table & Startup Code

On ARM Cortex-M, the processor reads two words from the beginning of Flash at reset: the initial stack pointer, and the address of Reset_Handler.

/* From main.c — vector table placed in .isr_vector section */
__attribute__((section(".isr_vector")))
const uint32_t vector_table[] = {
    _estack,                          /* 0: initial stack pointer (top of SRAM) */
    (uint32_t)Reset_Handler,         /* 1: reset vector */
    (uint32_t)Default_Handler,       /* 2: NMI */
    (uint32_t)Default_Handler,       /* 3: HardFault */
    /* ... */
};

void Reset_Handler(void) {
    /* 1. Copy .data from Flash to SRAM */
    uint32_t *src = &_sidata, *dst = &_sdata;
    while (dst < &_edata) *dst++ = *src++;

    /* 2. Zero .bss in SRAM */
    for (uint32_t *p = &_sbss; p < &_ebss; ) *p++ = 0;

    /* 3. Call main */
    main();
}
Analogy

The vector table is the phone directory the processor reads when it wakes up. Entry 0 = "where's the top of stack?" Entry 1 = "who do I call first?" The Reset_Handler is the opening act — move furniture (copy .data), vacuum the floor (zero .bss), then let main() run.

9. Linker Script

The linker script defines the memory map and tells the linker where to put each section.

/* linker/stm32f407.ld */
MEMORY {
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 1024K
    SRAM  (rwx) : ORIGIN = 0x20000000, LENGTH = 128K
}

SECTIONS {
    .text : {                        /* code goes to Flash */
        KEEP(*(.isr_vector))         /* vector table MUST be first */
        *(.text*)
    } > FLASH

    .data : {                        /* initialized variables */
        _sdata = .;
        *(.data*)
        _edata = .;
    } > SRAM AT> FLASH              /* stored in Flash, runs from SRAM */

    .bss : {                         /* zero-initialized globals */
        _sbss = .;
        *(.bss*)
        _ebss = .;
    } > SRAM
    _estack = ORIGIN(SRAM) + LENGTH(SRAM);  /* top of stack */
}

10. Ring Buffer Pattern

Used in both the FIR delay line and the sine LUT phase accumulator. A ring buffer wraps around using a power-of-2 mask instead of a branch.

#define SIZE 16      /* must be power of 2 */
#define MASK (SIZE-1) /* = 0xF */

uint8_t head = 0;
int16_t buf[SIZE];

/* Write: */
buf[head] = value;
head = (head + 1) & MASK;  /* 0→1→…→15→0, no branch */

/* Read N samples back: */
buf[(head - N) & MASK];     /* modular arithmetic, no bounds check */

11. Producer-Consumer (Lock-Free)

The ping-pong buffer implements a lock-free producer-consumer where the ISR is the producer (signals "buffer is done playing, swap it") and the main loop is the consumer (fills the next buffer).

  • Single writer to active flag (the ISR) → no race on write
  • Main loop reads active to pick which half to fill → read is always consistent
  • No mutex needed because the data structures are naturally partitioned

static I2SStatus wait_flag_clear(volatile uint32_t *reg, uint32_t mask) {
    uint32_t timeout = 100000;        /* countdown, not time-based */

Key Loops in pdm_filter_process_byte

/* Outer: iterate over 8 bits of the input byte (MSB first) */
for (int bit = 7; bit >= 0; bit--) {
    int32_t x = (pdm_byte >> bit) & 1;  /* extract 1 PDM bit */

    /* CIC integrators — 3 cascaded running sums */
    f->integrator[0] += x;
    f->integrator[1] += f->integrator[0];
    f->integrator[2] += f->integrator[1];

    if (++f->bit_count == CIC_DECIMATION) {   /* every 64 bits */
        f->bit_count = 0;

        /* CIC combs — 3 differentiators */
        int32_t v = f->integrator[2];
        for (int s = 0; s < CIC_STAGES; s++) {
            int32_t tmp = v - f->comb_prev[s];
            f->comb_prev[s] = v;
            v = tmp;
        }

        /* Scale CIC output (gain = 64^3 = 262144) → int16 range */
        int32_t cic_out = v / (1 << 9);   /* >>9 keeps headroom */

        /* FIR: insert into ring buffer, compute dot product */
        f->delay[f->delay_head] = (int16_t)cic_out;
        int32_t acc = 0;
        for (int k = 0; k < FIR_TAPS; k++) {
            acc += (int32_t)fir_coeffs[k]
                 * f->delay[(f->delay_head - k) & 0xF];
        }
        f->delay_head = (f->delay_head + 1) & 0xF;
        *pcm_out = (int16_t)(acc >> 15);  /* Q15 → int16 */
        return 1;                          /* sample ready */
    }
}
return 0;  /* need more bits */

6. Replacing DMA with Interrupt-Driven Transfer

/* I2S TX empty interrupt — fires 44100 × 2 = 88200 times/second */
void SPI2_IRQHandler(void) {
    if (SR & TXE) {
        *DR = *sample_ptr++;
        if (sample_ptr >= end_ptr) {
            sample_ptr = start_ptr;
            /* signal main loop to refill */
        }
    }
}

At 44.1 kHz stereo this ISR fires 88,200 times/second. Context switch overhead alone (~50 cycles each) = 4.4M cycles/s = 2.6% of a 168 MHz CPU. Acceptable for simple apps, but leaves no room for DSP, Bluetooth, or display updates. DMA drops this to ~94 ISRs/second.

8. FPGA Alternative

For the highest throughput or custom sample rates, the I2S controller and PDM filter can be implemented in FPGA logic (VHDL/Verilog).

  • CIC filter maps directly to shift registers + adders — purely combinational, no clock overhead
  • FIR filter implemented as systolic array — all 16 multiplications happen in parallel
  • Latency drops from microseconds to nanoseconds
  • Used in: professional audio interfaces, SDR (software-defined radio), MEMS mic arrays

Interview Preparation

Questions from real embedded/audio/systems interviews at Qualcomm, ARM, Apple, Texas Instruments, and audio DSP startups — answered from this codebase.

DMA & Interrupts

Q: Why use DMA instead of CPU-driven I/O for audio output?

At 44.1 kHz stereo, interrupt-per-sample means 88,200 ISR entries per second. Each context switch is ~50–100 cycles on Cortex-M4. That's 4–8 million cycles/second of overhead alone — not counting the actual data write. DMA moves entire buffers (512 samples) and fires one interrupt at completion. Overhead drops by ~512×. The CPU is free 99.9% of the time for DSP, display, or communication tasks.

This project proves the point: the DMA ISR is four lines — swap a pointer, done. No data copying in interrupt context.

Q: What is DMA circular mode and why is it used here?

In circular mode, when the DMA reaches the end of the configured buffer it automatically wraps back to the start and continues without CPU intervention. The DMA controller fires an interrupt at the halfway point (half-transfer) and at the end (transfer complete). This maps perfectly to ping-pong buffering: CPU fills the top half while DMA plays the bottom, then they swap.

Without circular mode, you'd have to reprogram the DMA source address in every ISR — adding latency and the possibility of missing a buffer handoff.

Q: What is a DMA underrun and how do you detect/handle it?

An underrun occurs when DMA tries to play from a buffer that the CPU hasn't finished filling yet. The result is silence, a glitch, or repeated stale samples — all audible as a pop or dropout.

In this driver: AudioBuf.underruns is a counter incremented in the ISR if the CPU hasn't updated the write-side pointer. Detection: check the underruns counter in the main loop. Prevention: ensure sine_gen_fill() completes faster than one DMA buffer period (512 samples / 44100 Hz ≈ 11.6 ms). Our fill is a table lookup — takes ~10 µs.

Q: Explain the difference between a half-transfer and transfer-complete DMA interrupt.

In DMA circular mode with a buffer of N items: the half-transfer interrupt (HTIF) fires when N/2 items have been transferred; the transfer-complete interrupt (TCIF) fires at item N. This gives you two interrupts per buffer cycle. In ping-pong: HTIF = "DMA just finished the first half, CPU can now fill it"; TCIF = "DMA just finished the second half, CPU can now fill it." Each interrupt lets you safely write to the half DMA just vacated.

Ping-Pong Buffers & Synchronisation

Q: Why don't you need a mutex for the ping-pong buffer swap?

Two reasons: (1) Only one writer to active: the ISR is the only code that calls audio_buf_swap(), which does active ^= 1. The main loop only reads active. Single writer + no torn write = no race. (2) The XOR of a single byte is atomic on ARM — the processor cannot be interrupted mid-byte-write because a byte write is a single bus transaction.

A mutex would add latency in the ISR and could cause priority inversion — never use mutexes in ISR context in real-time audio.

Q: What is the volatile keyword for and why is it critical in this driver?

volatile tells the compiler "this variable can change outside the C program's control — do not cache it in a register, do not optimize away accesses, do not reorder reads/writes across sequence points." In this driver it's used for: (1) hardware registers — the I2S status register's TXE bit is set by hardware; the compiler can't see that; (2) ISR-shared variables — active flag is written by the ISR and read by the main loop; without volatile the compiler might read it once and cache the result forever.

Analogy

Volatile is like a "refresh" button on a live scoreboard. Without it, the C compiler might display the score from when you first looked. With volatile, it checks the real scoreboard every time.

DSP — CIC & FIR Filters

Q: Why use a CIC filter instead of a direct FIR for PDM decimation?

The PDM bit rate is 3.072 MHz. A 64-tap FIR running at 3 MHz would need 64 multiply-accumulate operations per input bit = 196 million MACs/second. At 168 MHz with single-cycle multiply, that's 117% of CPU capacity — impossible. A CIC filter uses only addition and subtraction. 3 integrators, running every bit = 3 adds × 3.072M = 9.2M additions/second. The Cortex-M4 handles this trivially. The CIC output at 48 kHz is then cleaned up by the 16-tap FIR, which costs only 768K MACs/second.

Q: What is Q15 fixed-point and how do you multiply two Q15 numbers?

Q15 stores fractional values in a 16-bit signed integer. The implied binary point is after bit 15: value = integer / 32768. Range: -1.0 to +0.99997. To multiply two Q15 values: result = (int16_t)(((int32_t)a * b) >> 15). The multiplication of two 16-bit Q15 numbers gives a 32-bit Q30 result; right-shifting by 15 gives Q15 back. You must widen to int32 first to prevent overflow.

Example: 0.5 × 0.5 = 0.25. In Q15: 16384 × 16384 = 268,435,456 (Q30). >> 15 = 8192 = 0.25 in Q15. ✓

Q: Why are the FIR coefficients symmetric? What does that guarantee?

A symmetric (linear-phase) FIR filter has the property that all frequencies are delayed by the same amount (constant group delay). This means the filter introduces no phase distortion — transients and harmonics arrive at the output at exactly the same relative time as in the input. In audio this matters: phase distortion can make drums sound "smeared" or cause stereo image collapse. The 16 coefficients in this driver are symmetric around the center: coeffs[0] == coeffs[15], coeffs[1] == coeffs[14], etc.

Startup & Memory

Q: Walk me through what happens from power-on to the first audio sample.

1. CPU comes out of reset, reads stack pointer from address 0x08000000 (= top of SRAM = 0x20020000). 2. CPU reads reset vector from 0x08000004 → jumps to Reset_Handler. 3. Reset_Handler copies .data section from Flash to SRAM (initialized globals). 4. Reset_Handler zeroes .bss section in SRAM (uninitialized globals). 5. Calls main(). 6. main() initializes AudioBuf (memset to silence), I2SDriver (configures I2SCFGR, I2SPR registers), SineGen (builds LUT). 7. Pre-fills two half-buffers with 440 Hz sine data. 8. Calls i2s_start() which sets CR2.TXDMAEN — DMA begins streaming. 9. DMA copies from ping buffer to I2S DR → I2S peripheral clocks bits out on the wire → DAC converts to analog → speaker makes sound.

Q: What is the difference between .text, .data, .bss, and .rodata?

.text — executable code. Stored in Flash. Read-only at runtime. .rodata — read-only data (string literals, const arrays like the FIR coefficient table). Stored in Flash. .data — initialized global/static variables. Stored in Flash (the initial values), copied to SRAM at startup. .bss — uninitialized or zero-initialized global/static variables. Not stored in Flash — just a size. Zeroed in SRAM at startup. The linker script defines symbols marking the boundaries of each section, which Reset_Handler uses to know what to copy and what to zero.

Q: Why is the vector table placed at 0x08000000 and marked KEEP in the linker script?

ARM Cortex-M reads the vector table from the VTOR (Vector Table Offset Register) which defaults to 0x00000000 at reset. On STM32, address 0x00000000 is aliased to 0x08000000 (Flash) via the BOOT0/BOOT1 pins. If the vector table were not the very first thing in Flash, the reset vector would point to garbage and the CPU would crash immediately. KEEP(*(.isr_vector)) prevents the linker from garbage-collecting the vector table when it detects (incorrectly) that nothing "calls" it — the CPU references it directly via hardware, which the linker can't see.

System Design Questions

Q: How would you add reverb or EQ to this audio pipeline?

Insert a DSP stage between sine_gen_fill() and the buffer handoff. For EQ: implement a biquad IIR filter (5 coefficients per band, 2 state variables) in Q15. Each band costs 5 MACs per sample. For reverb: a Schroeder reverb uses 4 comb filters + 2 all-pass filters in a feedback topology. State is stored in ring buffers (delay lines of 10–100 ms). At 44.1 kHz with 100 ms delay lines: 4410 samples × 4 bytes × 4 combs ≈ 70 KB of SRAM — fits in our 128 KB SRAM. The key constraint is that all DSP must complete within one buffer period (11.6 ms).

Q: How would you scale this to 8-channel surround sound?

Switch from I2S (2 channels) to TDM (Time-Division Multiplexed) mode — same I2S bus, but WS toggles every 8 slots instead of every 2. The I2SCFGR register supports this on STM32. The AudioBuf would become 8 interleaved channels instead of 2. The DMA transfer size multiplies by 4. Main challenge: 8× the buffer memory and 8× the DSP compute. At 44.1 kHz × 8 channels × 16 bits × 512 frames = 8 MB/s DMA bandwidth — well within the 168 MHz AHB bus capability.

Q: What would change if you moved this to a Cortex-M0 instead of M4?

Three things: (1) No hardware multiplier (single-cycle MUL on M4 → 32-cycle on M0) — the FIR's 16 MACs per sample would cost 512 cycles instead of 16. At 48 kHz that's 24M cycles/second = 24% of a 100 MHz M0 just for the FIR. (2) No SIMD/DSP instructions — M4 has SMAD16, SMLAL etc. (3) No FPU — but we already use Q15 for this reason. The CIC is unaffected (adds only). The FIR would need to use __int64 accumulators and be careful about overflow. This is exactly why we chose Q15: the same code runs on M0 without modification.

Q: How do you debug audio glitches in a bare-metal system with no OS?

Four techniques: (1) Underrun counter — AudioBuf.underruns tells you if the main loop is too slow. (2) GPIO toggle — set a GPIO pin at ISR entry, clear at exit. Probe with oscilloscope. If the period is irregular, you have timing jitter. (3) Scope the I2S lines — WS should be a perfect square wave at 44.1 kHz; BCLK should be steady. Missing edges indicate a DMA underflow. (4) ITM/SWO trace — Cortex-M4 has the Instrumentation Trace Macrocell; you can printf() to a debug pin at near-zero overhead. Most glitches are caused by: ISR taking too long, main loop doing blocking work (flash writes, I²C waits), or DMA misconfiguration (wrong transfer count).
